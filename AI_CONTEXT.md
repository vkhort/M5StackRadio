# AI Context: M5StackRadio (Classic Core / M5Unified)

## 🎯 Глобальная цель проекта
Продвинутый многопоточный интернет-радиоприемник на базе контроллера M5Stack Core (Basic/Gray/Fire) с физическими кнопками (ESP32, 240 МГц) на фреймворке Arduino с использованием FreeRTOS, файловой системы SPIFFS и экосистемы M5Unified.

## 🔄 Жизненный цикл и Логика инициализации
При включении устройства строго соблюдается следующий алгоритм:
1. **Монтирование FS**: Класс `WorkSPIFFS` монтирует память SPIFFS и проверяет наличие текстового файла конфигурации `/config` (формат: `ключ=значение`).
2. **Проверка конфига**: Если файла `/config` нет, метод `setDefaults()` генерирует его «на лету», записывая константы по умолчанию из `config.h` (SSID, Password, стартовая громкость, станции).
3. **Чтение данных**: Из файла `/config` в оперативную память загружаются параметры текущей сессии (`ConfigData`) и динамический массив станций.
4. **Инициализация Wi-Fi**: Класс `WiFiConnect` пытается подключить устройство к роутеру, используя считанные SSID и пароль (таймаут 15 секунд).
5. **Разветвление режимов (STA / AP)**:
   - **STA Режим (Успех)**: Если Wi-Fi подключен, запускается синхронизация времени по NTP, поднимается асинхронный веб-сервер и стартуют 4 параллельные задачи FreeRTOS. Радио начинает играть.
   - **AP Режим (Отказоустойчивость)**: Если Wi-Fi не найден, M5Stack поднимает собственную скрытую точку доступа (SSID/PASS берутся из `/config`). Аудиопоток не воспроизводится. Устройство переходит в режим ожидания: пользователь может зайти через браузер на IP-адрес платы, чтобы перенастроить Wi-Fi или отредактировать список станций.

## 🏗 Архитектура задач FreeRTOS (STA Режим)
Межъядерное взаимодействие (Core 1 → Core 0) осуществляется строго через потокобезопасные атомарные очереди `xQueue`.
- **Core 0 (Аудиопоток)**: `AudioTask` (приоритет 5, стек 16Кб) — непрерывное декодирование MP3/HTTP потоков через библиотеку `ESP32-audioI2S` и заполнение I2S буфера. При наличии аппаратной PSRAM сетевой буфер выделяется в ней.
- **Core 1 (Интерфейсы и периферия)**: 
  - `ControlTask` (приоритет 2, 50 Гц) — опрос трех физических кнопок через `M5.update()` и сдвиг бегущей строки метаданных на экране ILI9341 (320x240 через `M5GFX`).
  - `GyroTask` (приоритет 2, 50 Гц) — параллельный опрос гироскопа MPU6886, фильтрация шумов (Alpha 0.3f, медиана 5) и распознавание жестов наклона (управление дублирует кнопки).
  - `NetworkTask` (приоритет 1) — обслуживание асинхронных запросов от веб-интерфейса браузера (`mathieucarbou/ESPAsyncWebServer`). Из браузера можно управлять плеером, громкостью и динамически менять URL потоков в памяти.

## ⚙️ Аппаратная спецификация и Логика Кнопок
- **Кнопки управления (M5Unified Style)**:
  - `M5.BtnA` (Левая): Клик — уменьшить громкость (`CMD_VOLUME_DOWN`) / Удержание (600мс) — предыдущая станция (`CMD_STATION_PREV`).
  - `M5.BtnB` (Средняя): Клик — запуск/остановка потока (`CMD_TOGGLE_PLAY`).
  - `M5.BtnC` (Правая): Клик — увеличить громкость (`CMD_VOLUME_UP`) / Удержание (600мс) — следующая станция (`CMD_STATION_NEXT`).
- **Звуковой тракт**: Встроенный динамик (GPIO25) через базовый усилитель NS4148 (Пины I2S: BCLK=12, LRC=13, DIN=15).
- **Кнопки управления (M5StickC Plus)**:
   - `BTN_A_PIN` (Лицевая кнопка M5): Клик — запуск/остановка потока (`GYRO_CMD_SHAKE` / `CMD_TOGGLE_PLAY`).
   - `BTN_B_PIN` (Боковая кнопка): Клик — следующая станция (`CMD_STATION_NEXT`).
   - `BTN_C_PIN` (Вторая боковая кнопка): Клик — увеличение/уменьшение громкости.
- **Жесты гироскопа (GyroJoystick)**:
   - Встряхивание (`GYRO_CMD_SHAKE`) — Воспроизведение / Пауза.
   - Крен (Roll Up/Down) — Управление громкостью.
   - Тангаж (Pitch Fwd/Bwd) — Переключение станций.
- **Звуковой тракт**: Пины I2S (BCLK=26, LRC=0, DIN=25).

## ⛔ Ограничения и Правила для ИИ (КРИТИЧЕСКИ ВАЖНО)
1. **Правило ссылок**: ИИ запрещено присылать ссылки на репозитории GitHub одной строкой (они обрезаются интерфейсом). Любая ссылка (например, на библиотеки) должна разбиваться строго на ДВЕ строки.
   *Пример:*
   https://github.com/
   mathieucarbou/ESPAsyncWebServer.git
2. **Асинхронность**: Категорический запрет на использование блокирующего метода `delay()`. Все паузы внутри задач FreeRTOS реализуются через `vTaskDelay`.
3. **Код интерфейса**: Использовать только современные методы `M5.update()`, `M5.Display` и `M5.BtnX` из экосистемы `M5Unified`.
4. **Изоляция графики**: ИИ категорически запрещено писать или генерировать методы отрисовки интерфейса (координаты, шрифты, цвета, очистку зон) внутри файлов задач FreeRTOS или главного скетча. Все изменения интерфейса делаются СТРОГО внутри методов класса `Display`. Задачи FreeRTOS взаимодействуют с экраном исключительно через публичный интерфейс класса `Display` (например: `display.drawTime()`, `display.updateStationName()`).

## 🗂 Структура проекта (Архитектурное разделение)
Проект построен по строгому модульному принципу. Перекрестное управление периферией из разных файлов запрещено:

1. **Главный файл управления (`src/M5StackRadio.ino`)**:
   - Точка входа в программу. Инициализирует глобальные объекты (`Display`, `WorkSPIFFS`, `WiFiConnect`, `Radio`).
   - Настраивает базовую конфигурацию чипа через `M5Unified` в функции `setup()`.
   - Проверяет целостность файловой системы, разворачивает логику сети (STA/AP сессии), собирает асинхронную HTML-страницу и запускает планировщик FreeRTOS.
   - Системный цикл `loop()` полностью разгружен (`delay(10)`) для обеспечения максимального кванта времени рабочим задачам.

2. **Конфигурационный манифест сборщика (`platformio.ini`)**:
   - Настройки компилятора под плату `m5stack-core-esp32`.
   - Включает оптимизацию кэша и поддержку PSRAM для организации больших аудиобуферов.
   - Перенаправляет папки кэша (`.pio-cache`) во избежание износа диска SSD.
   - Декларирует внешние зависимости (`M5Unified`, `M5GFX`, асинхронный сервер).

3. **Глобальный конфигуратор (`lib/vk_lib/config.h`)**:
   - Хранит жесткие дефайны аппаратной геометрии экрана (320x240), тайминги опроса задач, приоритеты FreeRTOS, размеры стеков и стартовый хардкод-список URL потоков.

4. **Изолированный графический движок (`lib/vk_lib/display.h`, `display.cpp`)**:
   - **Единственная точка взаимодействия с экраном.** Реализует класс `Display`.
   - Инкапсулирует внутри себя все приватные координаты (`_X`, `_Y`, `_W`, `_H`), размеры шрифтов, цвета и кэш-буферы для предотвращения мерцания (анти-фликкер подложки).
   - Задачи FreeRTOS и главный скетч взаимодействуют с дисплеем *исключительно* через абстрактные публичные методы (`display.drawTime()`, `display.updateVolume()`). Прямой вызов графических команд (`M5.Display.*` или `M5.Lcd.*`) за пределами этого класса строго запрещен.

5. **Ядро системы и Задачи (`lib/vk_lib/radio.h`, `radio.cpp`)**:
   - Разворачивает многопоточность через `xTaskCreatePinnedToCore`.
   - Обслуживает очереди `xQueue` для безопасной межъядерной связи (Core 1 ➔ Core 0).
   - Внутри задачи `ControlTask` обрабатывает физические кнопки M5Stack по схеме: короткий клик — громкость (`M5.BtnX.wasClicked()`), длинный клик (600мс) — переключение станций (`M5.BtnX.wasHold()`).

6. **Сетевой и Файловый стек (`lib/vk_lib/classes.h`, `classes.cpp`)**:
   - Реализует класс `WorkSPIFFS` (менеджмент конфигурационного файла `/config` на Flash-памяти).
   - Реализует класс `WiFiConnect` (логика STA-подключения с автоматическим Fallback в режим изолированной точки доступа AP при потере несущей частоты роутера).
